Min Heaps¶

Min Heaps are a data structure that can be used for many different purposes, and which are a particularly efficient mechanism for implementing priority queues.

If there is no balance discipline applied to the tree, it could in the worst case provide only $O(n)$ searching, but if the tree is balanced (through the AVL or some other balancing algorithm), then searching it for any item is guaranteed to be be $O(log_2 n)$.

Min heaps¶

Let's turn our attention now to a new kind of binary tree -- the min heap. There is also such a thing as a max heap, which is exactly a min heap with appropriate modifications. Min heaps and max heaps collectively are known as "binary heaps."

Note that in programming we often talk about "the heap" as a place in memory while data is stashed during running of programs (this will be a big topic in CS143); "the heap" is not a min heap or max heap and actually these kind of heaps have essentially no relationship. Just be aware that in talking about programming the word "heap" will crop up in different contexts with much different meanings.

A min heap has a constrained construction, like a BST, but less so. The requirement for a min heap is that every node is less than or equal to its two children (and this property is true at every node). For example:

By convention, a min heap is a complete binary tree (as this one is), but it would be possible to satisfy the min heap constraint even in a non-complete binary tree.

Notice how, with a min heap, searching becomes a $O(n)$ operation in general. For example, if we wanted to check whether the heap above contains 512, we would have to check every element before concluding it is not contained in the heap. However, we do have excellent searching guaranteed for one task in particular: finding the smallest value in the heap! (i.e., min finding) That's immediate and thus $O(1)$.

So we have here a data structure that is great for exactly one specialized thing, and, otherwise, not so great.

Min Heap Implementation¶

When we code up a min heap, we don't usually code up a binary tree in any of the ways we've done previously (expression trees, BSTs, AVL trees, Tries). Rather, we exploit the regular shape of the tree to represent the items of the tree side-by-side in an array (a contiguous block of memory, not often called "array" in Python but called by that name generally in programming).

We place each item in min heap in the array one level at a time. For example, the heap above would be represented as follows:

In [1]:
heap = [2, 4, 8, 18, 8, 99, 12, 21]

Given the index i of any element in the heap, we can produce its parent and children with the following expressions:

  • parent(i) is (i - 1) // 2
  • left(i) is 2*i + 1
  • right(i) is 2*1 + 2

Let's define some functions for this and informally check they behave as expected:

In [3]:
def parent(i):
    return (i - 1) // 2

def left(i):
    return 2*i + 1

def right(i):
    return 2*i + 2
In [5]:
print(f"The parent of {heap[7]} is {heap[parent(7)]}")

The parent of 21 is 18
In [4]:
print(f"The left child of {heap[2]} is {heap[left(2)]}")

The left child of 8 is 99
In [6]:
print(f"The right child of {heap[1]} is {heap[right(1)]}")

The right child of 4 is 8

Because we are able to represent the tree in an array, we are able to traverse the tree simply by hopping around the array according to these numerical relationships.

For heap insertion, we always fill in the next slot in the array, and then sift up by swapping items in the array logarithmically many times.

Similarly, for min removal, we swap the item from the last position in the tree into position 0, and then sift down by swapping items in the array logarithmically many times.

Priority Queue Revisited¶

Now that we have a min heap implementation, we can re-implement our priority queue to use it. Our code would look something like this:

In [2]:
from minheap import MinHeap

class PriorityQueue:

    _mh: MinHeap

    def __init__(self):
        self._mh = MinHeap()

    def enqueue(self, value: str, priority: int) -> None:
        self._mh.insert(priority, value)

    def dequeue(self) -> tuple[str, int]:
        elem = self._mh.remove_min()
        assert elem is not None
        prio, val = elem

        return val, prio

    def update_priority(self, value: str, new_priority: int) -> None:
        self._mh.change_priority(value, new_priority)

    @property
    def size(self) -> int:
        return self._mh.size

    def __str__(self) -> str:
        return str(self._mh)

Heapsort - [Optional]¶

This data structure lends itself to an efficient sorting algorithm called Heapsort, which is as follows:

Given an array of ordered items (WLOG integers):

  • construct a min heap by inserting items (including sifting up) as sketched above

  • as long as the heap is not empty, remove and emit the min, repairing the heap with sifting down after each removal

Heapsort can be, and is often, done entirely in an array without any “tree programming” per se. Operationally, it’s just a series of array swaps, but the understanding of that array as a min heap, along with sift up and sift down operations, is the best way to guide the programmer to get it right.

Given $n$ items to sort, heapsort does $n$ insertions (each taking $O(\log n)$ time) to construct the heap and $n$ removals (each taking $O(\log n)$ time) to deconstruct it once constructed. It requires $O(n\cdot \log n)$ in the construction phase and $O(n\cdot \log n)$ in the deconstruction phase. In the world of algorithm analysis, this makes it an $O(n\cdot \log n)$ algorithm altogether (even though it's two $O(n\cdot \log n)$ processes in sequence -- we don't care that it's twice). For sorting, that's really fast! You'll learn all this in detail in algorithms; this is just a preview of that curriculum.

In conclusion, heapsort is a case of "trees in your head, arrays in your code" in that trees aren't directly present in your code, despite their critical presence conceptually.
